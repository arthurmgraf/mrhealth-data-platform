Visão Estratégica: Antes vs Depois

Transformação do processo manual para uma plataforma de dados automatizada

Contexto do Problema

A MR. HEALTH opera 50 unidades de restaurante no sul do Brasil (RS, SC, PR), cada uma gerando 2 arquivos CSV diários (pedidos e itens) a partir de sistemas POS legados. A consolidação desses 100 arquivos por dia era feita manualmente pela equipe de operações, consumindo horas de trabalho qualificado e introduzindo erros silenciosos nos dados que baseavam decisões executivas. Com a expansão planejada para novos estados, esse processo se tornaria completamente inviável.

Antes: Processo Manual

  • 100 CSVs/dia processados manualmente via copy-paste em Excel
  • Consolidação em planilhas sem versionamento ou auditoria
  • Erros silenciosos de copy-paste descobertos dias/semanas depois
  • Decisões executivas baseadas em dados defasados (D+1 a D+3)
  • Dados de vendas (CSV) desconectados de dados cadastrais (PostgreSQL)
  • Impossível reconstruir estado dos dados em data passada
  • Cada nova unidade = +2 CSVs/dia de trabalho manual adicional

Depois: Plataforma Automatizada

  • Ingestão event-driven: upload no GCS dispara pipeline automaticamente
  • Validação automática de schema, tipos, completude e deduplicação
  • Registros inválidos isolados em quarentena com relatório de erro
  • Dados consolidados em < 3 minutos após upload
  • Visão integrada: vendas + produtos + unidades + geografia
  • Data lake imutável com histórico completo, auditável e reconstruível
  • Arquitetura serverless que escala para 500+ unidades sem alterações

Princípios da Arquitetura

Orientada a Eventos

Eventos de upload de arquivos no GCS disparam automaticamente Cloud Functions. Sem polling, sem cron jobs. Processamento em tempo real com zero overhead de infraestrutura.

Serverless

Cloud Functions, BigQuery e GCS escalam automaticamente. Pague apenas pelo uso (que permanece no free tier). Sem servidores para gerenciar ou provisionar.

SQL-First

Todas as transformações escritas em BigQuery SQL. Sintaxe familiar para equipes de dados. Fácil de testar, depurar e manter. Sem necessidade de cluster Spark.

Otimizado para Custo

Projetado para o GCP Free Tier desde o primeiro dia. Particionamento, clustering e pré-agregações minimizam custos de consulta. $0,00/mês de custo operacional.

Buckets no Google Cloud Storage

Por que existem 3 buckets no projeto?

O projeto utiliza 3 buckets no Cloud Storage. Apenas o primeiro foi criado manualmente — os outros dois são gerados automaticamente pelo GCP ao implantar a Cloud Function de 2ª geração.

mrhealth-datalake-485810

Tipo: Criado manualmente (IaC)
Finalidade: Data lake do projeto — armazena todos os arquivos CSV de vendas, dados de referência e arquivos em quarentena
Estrutura raíz:

  • raw/csv_sales/ — CSVs diários das 50 unidades, organizados em subpastas por data (2026/) e lotes de teste (test/, final/)
  • raw/reference_data/ — Dados estáticos (produto, unidade, estado, país)
  • quarantine/ — Arquivos que falharam na validação de schema

Tamanho: ~1 MB (0,02% do free tier de 5 GB)

gcf-v2-sources-*

Tipo: Criado automaticamente pelo GCP
Finalidade: Armazena o código-fonte da Cloud Function
Conteúdo real: csv-processor/function-source.zip (7,7 KB) — pacote compactado com main.py e requirements.txt
Gerenciamento: Controlado pelo Cloud Functions — atualizado a cada deploy
Tamanho: 7,7 KB

gcf-v2-uploads-*

Tipo: Criado automaticamente pelo GCP
Finalidade: Artefatos de build gerados durante cada deploy da Cloud Function via Cloud Build
Conteúdo real: 8 arquivos .zip (~5,8 KB cada) — um artefato por deploy realizado, nomeados com UUIDs
Gerenciamento: Controlado pelo Cloud Functions — acumula histórico de builds
Tamanho: ~47 KB total

Resumo: Custo de Armazenamento GCS

Os três buckets juntos ocupam < 2 MB do limite de 5 GB gratuitos do GCP Free Tier (0,04%). Os buckets gcf-v2-* são infraestrutura gerenciada pelo GCP e não geram custo adicional. Apenas o bucket mrhealth-datalake-* é relevante para a plataforma de dados.

02_orders.sql

Entrada: bronze.orders → Saída: silver.orders

  • Deduplicação: ROW_NUMBER() OVER (PARTITION BY id_pedido ORDER BY _ingest_timestamp DESC) — mantém apenas o registro mais recente de cada pedido
  • Padronização: tipo_pedido → order_type (ONLINE/PHYSICAL) via CASE WHEN
  • Enriquecimento de datas: EXTRACT(YEAR, MONTH, DAY) + FORMAT_DATE('%A') para dia da semana
  • Campo calculado: items_subtotal = order_value - delivery_fee
  • Limpeza: TRIM() em endereço, NULL para lojas físicas

03_order_items.sql

Entrada: bronze.order_items → Saída: silver.order_items

  • Deduplicação: ROW_NUMBER() OVER (PARTITION BY id_item_pedido ORDER BY _ingest_timestamp DESC)
  • Integridade referencial: INNER JOIN bronze.orders — descarta itens órfãos (sem pedido correspondente)
  • Type casting: CAST(qtd AS INT64) para garantir tipo numérico inteiro
  • Campo calculado: total_item_value = quantity × unit_price
  • Limpeza: TRIM(observacao), strings vazias → NULL

01_reference_tables.sql

Entrada: 4 tabelas Bronze → Saída: 4 tabelas Silver

  • products: TRIM(nome_produto) + filtro WHERE id IS NOT NULL
  • states: TRIM(nome_estado) + filtro NULL
  • countries: TRIM(nome_pais) + filtro NULL
  • units (enriquecido): LEFT JOIN states → LEFT JOIN countries — unidade sai com nome do estado e país acoplados

Por que o custo é $0,00/mês?

O GCP oferece um Free Tier permanente (não é período de teste) para seus serviços principais. A escolha de cada serviço considerou o Free Tier como requisito:

  • Cloud Storage (5 GB grátis): O volume total de CSVs de 50 unidades × 30 dias ocupa ~1 MB — 0,02% do limite. Mesmo com 10x o volume, o uso seria de apenas ~10 MB
  • BigQuery Storage (10 GB grátis): Todas as camadas (Bronze + Silver + Gold) ocupam ~2 MB total — 0,02% do limite. BigQuery comprime dados automaticamente, maximizando eficiência
  • BigQuery Queries (1 TB/mês grátis): Consultas analíticas e transformações SQL consomem ~10 GB/mês — apenas 1% do limite. Particionamento e clustering reduzem bytes escaneados em até 100x
  • Cloud Functions (2M invocações/mês grátis): Com ~100 CSVs/dia × 30 dias = ~3.000 invocações/mês — 0,15% do limite. Inclui 400.000 GB-segundos de compute gratuito
  • Looker Studio (ilimitado e gratuito): Ferramenta de BI nativa do GCP, sem limites de dashboards, relatórios ou usuários

  • Implementação (22/25): Pipeline end-to-end funcional, 97,2% cobertura de testes, 13 scripts SQL de transformação, Cloud Function implantada, documentação abrangente. Por que 97,2% de cobertura? Detalhamento dos Testes

A cobertura de 97,2% (22 testes unitários com pytest) cobre o módulo de geração de dados (generate_fake_sales.py), que é o componente crítico de ingestão da plataforma. Os testes validam:

  • Integridade de schema: Todos os campos obrigatórios presentes com tipos corretos (datas, decimais, inteiros, strings)
  • Regras de negócio: Valores de pedido dentro de faixas realistas, tipos de pedido válidos (ONLINE/FÍSICA), status válidos (FINALIZADO/CANCELADO/PENDENTE)
  • Consistência referencial: IDs de unidade entre 1-50, IDs de produto entre 1-30, relação correta entre pedidos e itens de pedido
  • Distribuições estatísticas: Proporção realista entre tipos de pedido, distribuição de valores, volume de itens por pedido
  • Edge cases: Campos opcionais (endereço de entrega, taxa de entrega), valores nulos tratados corretamente, encoding UTF-8
  • 2,8% não coberto: Código de I/O (gravação em disco) e tratamento de exceções de sistema operacional, que são cobertos por testes de integração manuais

5 Servidores MCP (Model Context Protocol)

O que é MCP?

O Model Context Protocol (MCP) é um padrão aberto que permite que modelos de IA se conectem de forma segura a fontes de dados e ferramentas externas. Cada servidor MCP expõe capacidades específicas (busca web, consulta de documentação, web scraping) que os agentes podem invocar durante o desenvolvimento. Isso transforma o Claude Code de uma IA isolada em uma plataforma de desenvolvimento interconectada com acesso em tempo real a conhecimento externo.
